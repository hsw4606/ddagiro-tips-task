# 08. 정기 수집 자동화 — 스케줄러 (제로베이스)

> 목표: 수집→정제→저장→분석을 **매일 자동 실행**한다. MVP는 cron/APScheduler, 확장 시 Airflow/Dagster.
> 선행: [02 네이버](02_네이버_데이터랩_검색API.md)~[07 LLM](07_LLM_Claude_분류감성인사이트.md)

## 1. 먼저 — "하나의 작업"으로 묶기

스케줄러는 "스크립트 1개를 정해진 시각에 실행"하는 도구다. 그 전에 파이프라인을 함수 하나로 묶는다.
`src/jobs/daily.py`:

```python
import datetime as dt
from src.collectors import naver_search, naver_datalab
from src.storage import db
from src.analysis import classify

def run_daily():
    today = dt.date.today().isoformat()
    keywords = ["두바이초콜릿", "약과", "소금빵"]   # 실제론 DB의 watchlist 로드
    for term in keywords:
        kid = db.upsert_keyword(term)
        rows = naver_search.search("blog", term)          # 수집
        # 분류·감성(대량은 배치 권장 — 07 §5)
        pos = sum(1 for r in rows if classify.classify(r["title"]).sentiment == "positive")
        ratio = pos / len(rows) if rows else None
        db.save_metric(kid, today, "naver_search",
                       mention_count=len(rows), sentiment_pos=ratio)   # 저장
    print(f"[{today}] {len(keywords)}개 키워드 수집 완료")

if __name__ == "__main__":
    run_daily()
```

> 먼저 수동 실행으로 정상 동작 확인: `python -m src.jobs.daily`

## 2. 방법 A — cron (가장 단순, 상시 켜진 서버)

매일 새벽 4시 실행 예:

```bash
crontab -e
```

```cron
# 분 시 일 월 요일  명령
0 4 * * * cd /path/to/ddalgiro-trend && /path/to/.venv/bin/python -m src.jobs.daily >> logs/daily.log 2>&1
```

- 절대경로 사용(가상환경 파이썬 경로 포함).
- 로그를 파일로 남겨 실패 추적.

> 맥은 절전 시 cron이 안 도므로, 상시 켜진 서버/클라우드 인스턴스에서 운영 권장.

## 3. 방법 B — APScheduler (파이썬 프로세스 내 스케줄)

앱이 계속 떠 있는 경우(예: 웹 서버와 함께) 코드로 스케줄.

```bash
pip install apscheduler
```

```python
from apscheduler.schedulers.blocking import BlockingScheduler
from src.jobs.daily import run_daily

sched = BlockingScheduler(timezone="Asia/Seoul")
sched.add_job(run_daily, "cron", hour=4, minute=0)   # 매일 04:00
print("스케줄러 시작…")
sched.start()
```

## 4. 소스별 주기 차등 (권장)

- 뉴스·검색트렌드: 1일 1회
- SNS(유튜브 등): 6~12시간
- 백테스팅·스코어링 재계산: 1일 1회(수집 후)
APScheduler면 `add_job`을 여러 개, cron이면 여러 줄로 분리.

## 5. 실패에 강하게 (운영 필수)

- **에러 격리**: 키워드 1개 실패가 전체를 죽이지 않게 try/except + 로깅.
- **재시도**: 일시 오류(429/타임아웃)는 백오프 재시도.
- **알림**: 실패 시 Slack/이메일 통지(선택).
- **멱등성**: 같은 날 재실행해도 중복 적재 안 되게(06 스키마의 `ON CONFLICT` 활용).

## 6. 확장 — Airflow / Dagster (성장기 이후)

수십 개 작업의 의존관계·재시도·모니터링이 필요해지면 워크플로 오케스트레이터로 이전:

- **Airflow**: DAG로 작업 의존성 정의, 웹 UI로 모니터링/재실행.
- **Dagster**: 데이터 자산 중심, 타입·테스트 친화.

> MVP에선 과하다. cron/APScheduler로 충분히 검증한 뒤 도입.
