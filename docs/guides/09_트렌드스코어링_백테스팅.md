# 09. 트렌드 스코어링 + 백테스팅 (제로베이스)

> 목표: 자사 데이터에 의존하지 않고, 정답 라벨이 없는(운영 전) 상태에서 (1) **휴리스틱 트렌드 스코어 v1**을 코드로 만들고, (2) **과거 트렌드로 백테스팅**해 "성장기 진입을 며칠 전에 잡았는지" 검증한다.
> 선행: [06 저장](06_데이터저장_postgres_pgvector.md), [07 LLM](07_LLM_Claude_분류감성인사이트.md)
> 배경: `docs/02` §5.3(스코어링), `docs/04` §2(콜드스타트)
> ⚠️ **두 v1 구분**: 본 가이드 휴리스틱(언급 40·검색 30·호감 20·소스 10)은 *공개 데이터 전용 제로베이스 구현*이고, 픽미 공동연구 **PoC v1**(검색 30·SNS 25·감성 20·재구매 15·인구 10, 자사 데이터 포함 — [07 §1.1](07_픽미공동연구_분석방법론.md)·`docs/02 §5.3.1`)과는 **별개 모델**이다. 둘 다 '고도화 출발점'이며, 백테스트 *비교군*은 이와 별개인 단순 이동평균 임계(§6.1)다.

## 1. 왜 휴리스틱부터인가

학습 데이터(정답 라벨)가 없으므로 처음엔 ML이 아니라 **규칙 기반 스코어**로 시작한다. 운영하며 모인 "추천→채택→성과" 피드백이 쌓이면 그때 ML로 고도화.

## 2. 입력 신호 (06 daily_metrics에서)

키워드별 시계열:

- `mention_count` (언급/글 수)
- `search_ratio` (검색 관심도 0～100)
- `sentiment_pos` (긍정 비율 0～1)
- 소스 다양성(몇 개 소스에서 잡혔나)

## 3. 트렌드 스코어 v1 — 코드 (LLM 아님)

"절대량보다 가속도(증가율)"가 핵심. 모멘텀 + 감성 + 소스다양성 가중합.
`src/analysis/scoring.py`:

```python
import statistics as st

def growth_rate(series: list[float]) -> float:
    """최근값 대비 직전 구간 평균의 증가율. series: 오래된→최신 순."""
    if len(series) < 4:
        return 0.0
    recent = st.mean(series[-3:])          # 최근 3포인트
    base = st.mean(series[-7:-3]) or 1e-9  # 그 직전 4포인트
    return (recent - base) / base

def momentum(series: list[float]) -> float:
    """증가율의 변화(2차) — 가속 중인지."""
    if len(series) < 8:
        return 0.0
    g_now = growth_rate(series)
    g_prev = growth_rate(series[:-3])
    return g_now - g_prev

def trend_score(mentions: list[float], search: list[float], sentiment_pos: float, n_sources: int) -> float:
    """0~100 점수. 가중치는 백테스팅으로 조정."""
    g = growth_rate(mentions)
    m = momentum(search)
    score = (
        40 * _clip(g) +                    # 언급 증가율
        30 * _clip(m) +                    # 검색 모멘텀(가속)
        20 * (sentiment_pos or 0.5) +      # 호감도
        10 * min(n_sources / 4, 1.0)       # 소스 다양성(신뢰도)
    )
    return round(min(score, 100), 1)

def _clip(x: float) -> float:
    return max(0.0, min(x, 1.0))           # 음수/과대값 정리
```

## 4. 확산 단계 분류 (성장기 진입 = 추천 타이밍)

```python
def stage(mentions: list[float]) -> str:
    g = growth_rate(mentions)
    m = momentum(mentions)
    if g <= 0.05:               return "태동기"       # 아직 잠잠
    if g > 0.05 and m > 0:      return "성장기"       # ← 추천 타이밍!
    if g > 0.3 and m <= 0:      return "정점"         # 가속 둔화
    return "쇠퇴기"
```

> **"성장기 진입 직후"** 를 추천 신호로 본다(`docs/02` §5.3). 정점·쇠퇴기 진입은 진입 경고.

## 5. 백테스팅 — 과거 트렌드로 검증 (정답 만들기)

이미 흥망이 끝난 아이템의 과거 데이터를 모아, 우리 스코어가 "정점 전에 성장기를 잡았는지" 확인한다.

### 5.0 검증셋(정답 라벨) — 실제 트렌드 기반

아래는 흥망 시점이 공개 보도로 확인되는 디저트 트렌드다. **정점월(라벨)** 은 뉴스·검색량 기준 1차 추정치이며, 실제 백테스트에서는 §2(네이버 데이터랩) 재수집으로 일 단위 확정한다. **급등급락형 + 장기지속형 + 최신 사례**를 섞어, 모델이 (a) 정점 전에 성장기를 잡고 (b) 지속형에 헛신호를 내지 않는지를 동시에 본다.

| 아이템 | 부상 시작 | **정점(라벨)** | 쇠퇴/반감 | 검증 포인트 |
|--------|-----------|----------------|-----------|-------------|
| 탕후루 | 2023 상반기 | **2023-08** (브랜드검색 ～67.7만) | 반감기 54일, '23-10 전월비 −43% | 전형적 급등급락 — 리드타임·오탐의 기준 케이스 |
| 두바이 초콜릿 | 2024-04～05 | **2024-07 둘째 주** (카다이프 검색 정점) | 2025년 판매 −40% | 글로벌(’23-12 틱톡)→국내 전파 시차 |
| 요아정(요거트아이스크림) | 2024-06 | **2024-07～08** (검색 ～94.9만) | 2025년 출점 둔화·쇠퇴 | 탕후루 직후 계승, 커스터마이징형 |
| 약과(할매니얼) | 2022 | **2023 상반기** | 점진 안착(완전 소멸 아님) | 중기 트렌드형 — 정점 판정 난이도 |
| 베이글(런던베이글뮤지엄) | 2022 | **2022～2023** | 스테디셀러로 안착 | 핫플 기반 확산 |
| 소금빵 | 2021～2022 | **뚜렷한 정점 없이 스테디셀러화** | — | **반례(지속형)** — 성장기 헛신호 안 내는지 검증 |
| 두쫀쿠(두바이 쫀득쿠키) | 2025-12 | **2026-01** | 2026-02 중순 냉각 | 최신·초단기 — 거의 실시간 재현 검증 |

> 라벨 근거는 보도·검색량 기사(소믈리에타임즈·뉴스1·서울신문 등). 제안서/발표용으로는 **데이터셋·코드·결과표를 재현 가능하게 보관**해 제3자 공인에 대비한다(`docs/01 §3·§7`).

### 5.1 절차

1. 검증 대상 선정: §5.0 표(흥망 시점이 알려진 것).
2. 데이터 확보: 네이버 데이터랩 `search_trend(...)`로 과거 1～2년 추이를 일/주 단위 수집(02 가이드).
3. 시뮬레이션: 날짜를 하루씩 진행하며 그날까지의 데이터로 `stage()`를 호출, **언제 처음 "성장기"로 판정했는지** 기록.
4. 평가: 실제 정점 날짜 대비 **리드타임(며칠 앞섰나)** 과 오탐(헛신호) 측정.
`src/analysis/backtest.py`:

```python
from src.analysis.scoring import stage

def first_growth_signal(daily_mentions: list[tuple[str, float]]) -> str | None:
    """daily_mentions: [(date, value)] 오래된→최신. 처음 '성장기' 판정일 반환."""
    values = []
    for date, v in daily_mentions:
        values.append(v)
        if stage(values) == "성장기":
            return date
    return None

# 평가 예: 실제 정점일과 비교해 리드타임 계산
def lead_time(signal_date: str, peak_date: str) -> int:
    import datetime as dt
    s = dt.date.fromisoformat(signal_date); p = dt.date.fromisoformat(peak_date)
    return (p - s).days     # 양수 = 정점보다 그만큼 앞서 포착
```

### 5.2 검증셋 수집·실행 스크립트

§5.0 표를 그대로 코드화한다. 데이터랩은 **요청 기간 내 최댓값=100 상대비율**을 주므로, 아이템마다 *부상～쇠퇴를 포함한 자체 기간*으로 따로 수집하면 100=그 아이템의 정점이 된다(아이템 간 절대비교가 아니라 곡선 모양만 보므로 적합). 급등급락형 4종은 정점월 라벨이 뚜렷하고, **지속형(베이글·소금빵)은 정점 라벨을 비워(`None`)** 성장기 신호 오탐 여부만 본다.

`src/analysis/backtest_dataset.py`:

```python
from src.collectors.naver_datalab import search_trend          # guides/02 §4
from src.analysis.backtest import first_growth_signal, lead_time # 이 문서 §5.1

# (키워드, 수집 시작, 수집 종료, 라벨 정점월) — 09 §5.0 표와 1:1
VALIDATION_SET = [
    {"kw": "탕후루",          "start": "2023-01-01", "end": "2024-03-31", "peak": "2023-08"},
    {"kw": "두바이초콜릿",     "start": "2024-01-01", "end": "2025-06-30", "peak": "2024-07"},
    {"kw": "요아정",          "start": "2024-01-01", "end": "2025-06-30", "peak": "2024-08"},
    {"kw": "두바이쫀득쿠키",   "start": "2025-09-01", "end": "2026-06-15", "peak": "2026-01"},
    {"kw": "약과",            "start": "2021-06-01", "end": "2023-12-31", "peak": "2023-03"},  # 근사(할매니얼 상반기)
    {"kw": "런던베이글뮤지엄",  "start": "2021-06-01", "end": "2023-12-31", "peak": None},      # 지속 상승형
    {"kw": "소금빵",          "start": "2021-01-01", "end": "2024-06-30", "peak": None},      # 지속형 반례
]

def fetch_series(item) -> list[tuple[str, float]]:
    """데이터랩 주간 검색비율을 (기간, 값) 오래된→최신으로."""
    res = search_trend([item["kw"]], item["start"], item["end"], time_unit="week")
    return [(p["period"], float(p["ratio"])) for p in res[0]["data"]]

def empirical_peak(series) -> str:
    """실측 정점 = 검색비율 최댓값 날짜(라벨 타당성 교차검증용)."""
    return max(series, key=lambda x: x[1])[0]

def run():
    for item in VALIDATION_SET:
        series = fetch_series(item)
        signal = first_growth_signal(series)              # 처음 '성장기' 판정일
        emp = empirical_peak(series)
        if item["peak"] and signal:
            lt = lead_time(signal, f"{item['peak']}-15")  # 정점월 15일을 대표 정점일로
            verdict = f"리드타임 {lt}일" + ("  ⚠오탐(정점 이후 신호)" if lt < 0 else "")
        elif item["peak"] is None:
            verdict = "지속형 — 강한 성장기 신호가 없어야 정상(있으면 오탐)"
        else:
            verdict = "신호 없음"
        print(f"[{item['kw']:9}] 신호일={signal} 실측정점={emp} 라벨={item['peak']} → {verdict}")

if __name__ == "__main__":
    run()
```

실행: `python -m src.analysis.backtest_dataset`

#### 판독 가이드

- **리드타임 ≥ 14일(양수)**: 정점보다 그만큼 앞서 성장기를 포착 → KPI 충족([`docs/02` §10.1](../02_트렌드분석추천시스템_기획.md)).
- **실측정점 ≈ 라벨정점**: 데이터 품질·라벨 타당성 교차확인(크게 어긋나면 키워드 표기·기간 재점검).
- **베이글·소금빵(지속형)**: 성장기 신호가 안 떠야(또는 약하게만) 정상 — 헛신호 시 **오탐률**에 반영.
- 결과를 베이스라인(단순 7일 이동평균 임계)과 비교해 §6에서 가중치·임계값을 튜닝.

> **주의**: ① 데이터랩 키워드는 띄어쓰기에 민감 — `두바이쫀득쿠키`/`두바이 쫀득 쿠키` 등 표기 변형을 콘솔에서 확인해 채택. ② 데이터랩 과거 데이터는 2016-01부터 제공(검증셋 전 기간 커버). ③ `peak` 라벨은 보도 기준 근사 → 1차 실행 후 `empirical_peak`로 보정 가능.

## 6. 베이스라인 비교 & 가중치 튜닝

### 6.1 베이스라인(비교군) — "왜 R&D인가"의 핵심 증거

TIPS 심사는 *"단순 임계값이면 될 일을 왜 연구로 하나"* 를 묻는다. 그래서 우리 기법(§3～4)을 **단순 이동평균 임계** 베이스라인과 같은 검증셋에서 나란히 돌려 **리드타임 차이**를 보인다. (이 *단순 이동평균*이 개선폭을 재는 **비교군(naive baseline)** 이며, 휴리스틱 v1·픽미 공동연구 v1은 *고도화 출발점*이지 이 비교군이 아니다.)

`src/analysis/baseline.py`:

```python
import statistics as st

def baseline_signal(series, window=7, threshold=0.5):
    """단순 베이스라인: 직전 window 평균 대비 현재값이 threshold(50%) 이상
       급증한 첫 시점. series: [(period, value)] 오래된→최신."""
    values = []
    for period, v in series:
        values.append(v)
        if len(values) <= window:
            continue
        ma = st.mean(values[-window-1:-1])         # 직전 window 평균
        if ma > 0 and (v - ma) / ma >= threshold:
            return period                          # 첫 급증 시점
    return None
```

### 6.2 우리 기법 vs 베이스라인 일괄 비교

`src/analysis/compare.py`:

```python
from src.analysis.backtest_dataset import VALIDATION_SET, fetch_series  # §5.2
from src.analysis.backtest import first_growth_signal, lead_time        # §5.1
from src.analysis.baseline import baseline_signal                       # §6.1

def _lt(signal, peak):
    """정점월 라벨이 있을 때만 리드타임(일). 없으면 None."""
    return lead_time(signal, f"{peak}-15") if (signal and peak) else None

def run():
    rows = []
    for item in VALIDATION_SET:
        s = fetch_series(item)
        ours = _lt(first_growth_signal(s), item["peak"])
        base = _lt(baseline_signal(s), item["peak"])
        rows.append((item["kw"], item["peak"], ours, base))
        print(f"[{item['kw']:9}] 라벨={item['peak']}  우리={ours}  베이스={base}")

    labeled = [r for r in rows if r[1]]            # 정점 라벨 있는 급등급락형만 평균
    def avg(i):
        xs = [r[i] for r in labeled if r[i] is not None]
        return round(sum(xs) / len(xs), 1) if xs else None
    print(f"\n평균 리드타임 — 우리: {avg(2)}일 / 베이스라인: {avg(3)}일 "
          f"(개선 {round((avg(2) or 0) - (avg(3) or 0), 1)}일)")

if __name__ == "__main__":
    run()
```

실행: `python -m src.analysis.compare`
→ 출력의 **평균 리드타임 차이**가 곧 `docs/02` §10.1 결과 기록표의 "개선폭"이 된다.

### 6.3 가중치 튜닝 루프

비교 결과(리드타임↑·오탐↓)를 보고 §3 `trend_score`의 가중치(40/30/20/10)와 §4 임계값(0.05, 0.3 등)을 조정 → 여러 과거 아이템에서 평균 성능이 좋아지도록 반복. **베이스라인 대비 우위가 유지되는지**를 매 튜닝마다 확인.

## 7. v2(ML)로 가는 길

- 서비스 운영으로 "추천한 아이템이 실제로 떴는지" 라벨이 쌓이면, 이를 정답으로 한 **지도학습 모델**(예: 시계열 특징 + 그래디언트 부스팅)로 교체.
- LLM은 계속 "왜/누가/어떻게"의 **해석·브리핑**(07 §7) 담당. 점수 계산은 코드/ML.

## 8. KPI (성능 측정 지표)

- **리드타임**: 경쟁/일반 인지 대비 며칠 먼저 잡았나.
- **적중률**: "성장기" 판정 아이템 중 실제로 정점까지 간 비율.
- **오탐률**: 성장기로 판정했으나 뜨지 않은 비율.
