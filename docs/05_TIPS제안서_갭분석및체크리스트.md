> **목적**: `01~04`는 *기술·비용 기획* 문서다. 이 문서는 **"실제로 TIPS 일반(기본) 8억 트랙에 채택되려면 무엇이 더 필요한가"** 를 심사 관점에서 점검·추적하는 운영 문서다. 제안서 작성·제출 전까지 **갭(부족분)과 To-Do를 한곳에서 관리**한다.
>
> **사용법**: §3 갭표의 상태(🟢충분 / 🟡보강 / 🔴부족)를 주기적으로 갱신하고, §5 To-Do를 우선순위대로 닫는다. 결정/진척은 `PROGRESS.md`에도 반영.

## 4. 최대 리스크 — "그냥 LLM API 쓰는 거 아닌가?" 돌파법

정부 R&D는 **연구개발 과제**여야 한다. 현재 설계(수집→Claude 분류·요약→추천)는 자칫 **서비스 개발**로 비칠 수 있다. 아래를 제안서에 명확히 박아 **연구성**을 입증한다.

1. **난제 정의(왜 안 풀리나)**: "노이즈 많은 멀티소스 신호에서 *정점 이전 성장기*를 식별" / "복합 자연어 의도 → 하드제약 충족 랭킹" 은 단순 프롬프팅으로 정확도가 안 나오는 문제임을 baseline으로 제시.
2. **정량 목표(KPI)**: 예) *트렌드 조기예측 적중률 ≥ X%*, *경쟁 대비 리드타임 ≥ N일*, *추천 클릭/전환율 ≥ Y%*, *카탈로그 신선도(폐업 오탐) ≤ Z%*. 측정 방법·검증셋(백테스팅) 명시.
3. **자체 기법(LLM 위에 얹는 R&D)**: 시계열 확산(S-curve) 모델, 멀티소스 가중·신뢰도 융합, 도메인 특화 NER/신조어 탐지, 하이브리드 검색 + Learning-to-Rank, 환각 통제형(계산-생성 분리) 아키텍처 — *"LLM은 부품, 핵심은 우리 스코어링/랭킹/검증 알고리즘"* 으로 포지셔닝.
4. **데이터 해자**: 추천→채택→성과 피드백으로 축적되는 **고유 라벨링 데이터셋**이 복제 난도를 높인다(시간이 갈수록 강해지는 R&D 자산).

> 한 줄 메시지: **"우리는 LLM을 호출하는 회사가 아니라, 트렌드 예측·매칭의 정확도를 연구로 끌어올리는 회사다."**

## 6. 제출 전 최종 체크리스트

**서면(사업계획서)**
- [ ] 문제·솔루션·차별성 1페이지 요약(Executive Summary)
- [ ] 기술성/R&D 난제 + 정량 KPI (§4)
- [ ] 사업화·시장·수익모델·재무
- [ ] 팀 구성·역량
- [ ] 추진 일정·예산(8억 비목)
- [ ] 데이터·법적 거버넌스

**발표(대면)**
- [ ] 10~15분 피치덱(문제→기술→시장→팀→요청)
- [ ] 데모/PoC 1개
- [ ] 예상 질문 대비(특히 "이게 왜 R&D인가", "해자가 진짜 있나", "정확도 근거")

**증빙**
- [ ] 운영사 투자·추천 서류
- [ ] 팀 이력 증빙
- [ ] (있으면) LOI·백테스팅 결과·MVP
